Description

You are to write an assembler for the Stack Machine. The behavioral characteristics of the stack machine are described in the document assembler-doc. That document also explains the mnemonics for the stack machine opcodes and broadly describes the features the stack machine assembler provides.

Input to the stack machine assembler should be a text file containing lines of the following format: 

[ label [:]] [ operation [ operand ] ] [ ;comment ]
The items label, operation, operand, and ;comment are called fields. An item enclosed in brackets is optional. As a consequence, a blank line is a legal assembler line. Any combination of spaces and/or tabs (whitespace) may be used to separate the fields of a line; it is not necessary that whitespace precede a colon or semicolon. Each line must be terminated with a standard end-of-line mark.

An operation may be any of the stack machine opcode mnemonics, plus the following assembler directives: 

DS	DW	EQU	ENTRY	EXTERN	PUBLIC

Stack machine opcodes always generate executable code (an opcode, possibly followed by an operand value), while assembler directives are intended to direct assembler activities and do not in general generate executable code. Together, the opcode mnemonics and directives are reserved words. Reserved words may appear in uppercase, lowercase, or any mixture of cases.

A label is any sequence of letters, digits, dots, and underscores, beginning with a letter or at-sign '@' character, but excluding reserved words. A label is also called an identifier. Labels longer than 30 characters in length are silently truncated to 30 characters.

Labels that differ in case are considered to be different, so that XYZ and xyz are considered different labels.

A label beginning with an at-sign '@' character identifies it as a "local label". Internally, local labels are prepended with the most recently encountered non-local label definition (which defines the 'scope' of the local label) for the purpose of putting them in the symbol table and are otherwise treated as ordinary labels. This allows for defining simple labels within small code blocks (beginning with the definition of a non-local label) that do not conflict with the same local label names in other code blocks. Note that a non-local label defined in an EQU directive is not considered defining a new local label scope.

The start of an assembly language program is considered as beginning with an empty non-local label name. Local labels cannot be declared as ENTRY, PUBLIC, or EXTERN, although a non-local label can be EQUd to a local label and the non-local label then can be declared as PUBLIC or ENTRY. Moreover, an ENTRY, PUBLIC, or EXTERN label declaration does not define a new local label scope.

If a label is the first item on an input line and is followed by a colon ':', the assembler associates with it a value equal to the current location counter at the beginning of the line. If the label is followed by the EQU directive, the assembler associates with it a value equal to the operand expression that follows the EQU directive. In either case, the label is said to be defined when it appears as the first field on an input line, and such a label is said to be in label position. It is an error (multiply defined label) for a label to appear more than once in label position.

If an undefined label appears in an operand expression, the entire expression evaluates to an undefined value. Except for expressions in DS directives, undefined values in pass 1 will not result in an assembly error. In pass 2, the value of an operand expression that is undefined will not result in an assembly error unless the value is used to create an entry in the object module.

A label at the beginning of a line must be followed by a colon ':' except in the case of an EQU directive, in which case it must not be followed by a colon.

Operands may be ordinary algebraic expressions including the arithmetic operations of addition (+), subtraction (-), multiplication (*), division (/), and modulus (%). Unary minus (-) is also allowed. An operand within an expression may be either a label, the '$' symbol, an (unsigned) integer constant, or a character constant. Parentheses may be used to alter the normal arithmetic precedence rules.

Operands to multiplication, division, and modulus must be absolute, and the result is absolute. Operands to addition, subtraction, and unary minus may be absolute or relocatable. The result will be absolute or relocatable, depending on the absolute or relocatable attributes of its operands.

The value of a label in an expression is the value associated with the label at the time the expression is evaluated. A label's value may be either absolute or relocatable, depending on how the label is defined. A label may also be external (if defined in a EXTERN directive), in which case it is implicitly absolute.

The only permissible expressions involving EXTERN labels are EXTERN, EXTERN+abs, and EXTERN-abs. Internally, the assembler treats an EXTERN label as having an absolute value of zero for the purposes of generating object module code. Locations in the object module corresponding to an expression generated from an EXTERN label will be flagged in the object module in an EXTERN label entry and will be fixed up upon linking the module with another object module that has defined the label as PUBLIC. An EXTERN label cannot be declared as PUBLIC in the same source code.

The value of the special symbol '$' in an expression is the value of the location counter at the beginning of the line in which the expression occurs. This value is always relocatable.

An integer constant is a sequence of decimal digits '0', '1', ... '9' representing a decimal number. This number should not exceed the range of possible 32-bit signed integers, i.e., -2147483648 to 2147483647. (Your assembler is not required to check for this.) An integer constant is always absolute. An integer constant beginning with the digit '0' and followed by one or more octal digits (the digits '0' through '7') represents an octal number. An integer constant beginning with the digit '0', followed by either the letter 'x' or 'X', and followed by one or more hexadecimal digits (the digits '0' through '9' and the letters 'A' through 'F' in either upper or lower case) represents a hexadecimal number.

A character constant is any printable ASCII character (with ASCII values from 32 to 126 inclusive) preceded by a single quote mark; for example, 'F evaluates to the constant 70 and ' (quote followed by a space) evaluates to the constant 32. The single quote character itself is represented as ''. A character constant is treated exactly the same as its corresponding integer ASCII value, and can occur wherever an integer constant is permitted. A character constant is always absolute.

As an illustration of an expression, if the label LBL is relocatable and has value 2 and the location counter is 5, then the expression 

($-LBL)*'F
evaluates to absolute 210.

If a label in an operand expression is undefined at the time the expression is evaluated, the entire expression is considered undefined. In pass one, undefined expressions which are syntactically correct should not cause an error, except in DS directives, where the expression must evaluate to an absolute non-negative value. In pass two, undefined expressions always cause an error if they are operand expressions that generate object module code. Also at the end of pass two, labels that appear in PUBLIC or ENTRY declarations must have associated values that are defined and relocatable.

The location counter is initialized to zero when assembly begins. Each assembled instruction increments the location counter by the number of words required to form the instruction; instruction operations that require one operand increment the location counter by two, while instruction operations that require no operands increment the location counter by one. The value of the location counter is always considered relocatable.

The DS directive increments the location counter by the value of the operand expression. The operand expression must be defined in pass one of the assembler, must be non-negative, and must evaluate to an absolute--not relocatable--value.

The DW directive increments the location counter by one for each word required to store the operand list that follows the directive. The operand list has the form 

expr [ , expr ]*
where expr is an expression as defined above, or a string of printable characters enclosed in double quotes (").

In a string, the double quote character itself may be represented in a string if it is preceded by the escape character (\). Other standard escape sequences are also allowed in strings, as described below. While an escape sequence occupies two characters in the source string, the corresponding character in the token string that is used to generate operand words in the object module will consist of a single character value.

The following escape sequences should be recognized in a string: backspace (\b), tab (\t), newline (\n), formfeed (\f), return (\r), single quote (\'), double quote (\"), and the escape character itself (\\). The following table gives the character values of escape sequences that may appear in a string and that produce non-printing characters in the corresponding token string:
\b8
\t9
\n10
\f12
\r13
Quoted strings may not cross line boundaries.

The DW directive with a quoted string is identical to the exploded string obtained by replacing each character in the string with its ASCII character equivalent. for example, the directive 

 	DW	"this is a string\n"
is equivalent to the directive 
 	DW	't,'h,'i,'s,' ,'i,'s,' ,'a,' ,'s,'t,'r,'i,'n,'g,10
or even to the directive 
 	DW	116,104,105,115,32,105,115,32,97,32,115,116,114,105,110,103,10

Characters in a token string are deposited in the object module one character to a word, using the standard ASCII codes, in the order that they appear in the string. Successive values are stored in successively higher memory locations. There is no distinction between a string consisting of one character and the ASCII code for the character itself, so that "F" and 'F in a DW directive result in exactly the same allocation of storage. Similarly, "\n" and 10 result in exactly the same allocation of storage. The assembler does not insert a null character in the object module following the last character of a string.

Expressions used in DW directives may be undefined in pass one of the assembler, but must be defined in pass two. Expressions may evaluate to absolute or relocatable values. A quoted string and character constants are always stored as absolute (non-relocatable) ASCII values.

The EQU directive assigns the value of the operand expression to the associated label. A label must be present on an EQU directive line and must not be followed by a colon; in all other cases a label is optional. The value of the operand expression in an EQU directive may be undefined in pass one of the assembler, but must be defined (absolute, relocatable, or Extern) in pass two unless that label does not appear in any operand expressions that generate object module code or in any PUBLIC or ENTRY directive.

The EXTERN directive is followed by one or more non-local labels separated by commas. This directive declares the indicated labels as being defined in another module but used in this module. It is an error for an identifier to be both declared Extern and to appear in label position.

The PUBLIC directive is followed by one or more non-local labels separated by commas. This directive marks the indicated labels as being defined in this module and accessible in other modules through an EXTERN directive in those modules. All PUBLIC labels must be defined and relocatable at the end of pass 2.

The ENTRY directive is followed by one non-local label. This directive marks the indicated label as the entry point of any executable program linked with this module. An ENTRY label must be defined and relocatable at the end of pass 2.

A comment begins with a semicolon (;) and continues to the end of the line. Comments are not tokens. They are ignored for the purposes of generating object code, and they do not affect the location counter. A semicolon inside of a quoted string or in a character expression (';) is not considered as the beginning of a comment.

There should be a predefined label 'SP' with a predefined value of 0 (absolute), as if there were a line 

SP	EQU	0
at the beginning of every source code program. The lexical analyzer should consider all upper/lower case combinations of ''SP'' to be equivalent to the single token SP. This label may be used in expressions, but should be included in the cross-reference list if and only if it appears in the body of your program.

Output from the assembler is a text file in stack machine object module format as illustrated below and described in the next section.

%SXX+Object Module
text_length
% text
...
% relocation dictionary
...
% ENTRY, EXTERN, and PUBLIC declarations
...
% end of object module

Object Module Format

The format of an object module appears as follows: 

%SXX+Object
text_length
% text
...
% relocation dictionary
...
% ENTRY, EXTERN, and PUBLIC declarations
...
% end of object module
On lines beginning with a %, any characters may appear following the % on the line.

Text and relocation dictionary lines in the object module contain a single integer optionally preceded by whitespace and followed (preceded by whitespace) by any text up to end-of-line. The integer represents a data value or stack machine instruction. A line in the text portion of the object module may also consist of DS entry -- a colon ':' followed by a single integer, which has the effect of adding the integer value to the current location counter. This is useful for processing DS directives.

The text_length is the number of stack machine words determined by the text portion of the object module, adjusted appropriately for DS entries. The relocation dictionary contains the addresses of all object module words that are relocatable and is used when linking the object module to produce a Stack Machine executable.

An ENTRY line has the format 

ENTRY label addr
where label is the name of the entry point and addr is the code address in the object module which should begin execution when the module is linked and run. There can be at most one ENTRY line in an object module.

A PUBLIC line has the format 

PUBLIC label addr
where label is the name of the exported label and addr is its exported value (represented as a decimal integer). There can be zero or more PUBLIC lines in an object module.

An EXTERN line has the format 

EXTERN label addr addr...
where label is the name of the imported label and addr ... are whitespace-separated addresses (represented as decimal integers) where the symbol is referenced. There may be zero or more EXTERN lines in an object module.

The assembler is invoked using the 'sax' command, suggesting the 'Stack machine Assembler eXtended'. The format of the command line is 

sax [-o ofile] [-l lfile] file
If the optional '-o ofile' arguments are given, the object module is deposited in the file 'ofile'; otherwise the object module is sent to the standard output. If the optional '-l lfile' arguments are given, a listing of the assembled file is deposited in the file 'lfile'; otherwise no listing file will be produced. The 'file' argument must correspond to a text file in the assembly language format described above.

Listing File Format

The listing file should have the format described in the following example: 

 Line        LC Op     Operand	Source Line
 ----        -- --     -------	-----------
    1:        0             15  A       EQU     15
    2:        0              6R B       EQU     str + 3
    3:        0  2           6R START:  PUSHV   B
    4:        2 26                      HALT
    5:        3             65  str:    DW      "ABC",$
              4             66  
              5             67  
              6              3R 
    6:        7                 ; this is a comment
    7:        7            100  @1:     DS      100
  EOF:      107


Cross Reference List
--------------------
Symbol                                 Value   Line Numbers
------                                 -----   ------------
A                                         15 |     1*
B                                          6R|     2*    3 
START                                      0R|     3*
str                                        3R|     2     5*
@1                                         7R|     7*

The cross-reference list should be in alphabetical order by label, with the line number where the label is defined followed by an asterisk (*). All local labels associated with a preceding non-local label should appear in the cross-reference list following the non-local label, in alphabetical order by label.

In the cross-reference list, there should be no more than 80 characters per line; line numbers that do not fit on an 80 character line should wrap to the following line, aligned with the ones on the previous line.

A line of an object module beginning with the '#' character is considered a comment and is ignored by the linker. The first and last lines of an object module must not be comments.

Lexical Analysis

The lexical structure for the assembler is similar to that of Pascal, except that:

  • whitespace consists only of tabs and spaces -- except for quoted strings and single-character constants, tokens cannot contain whitespace;
  • comments are considered the same as whitespace -- they are not returned as tokens
  • there are different keywords as described below;
  • strings are enclosed in double quotes;

The following tokens should be recognized:

  • Keywords, i.e., all opcode mnemonics given in assembler-doc as well as the DS, DW, EQU, ENTRY, EXTERN, and PUBLIC directives.

  • The following special symbols: 
    +  -  *  /  %  $  ,  (  )  :

  • Label: any character string consisting of letters, digits and underscores (_), beginning with a letter or at-sign '@'. Only the first 32 characters of a label are significant.

  • Unsigned integer: any integer in octal, decimal, or hexadecimal format.

  • Single-character constant: any printable character preceded by a single quote (').

  • Quoted string: any sequence of printable ASCII characters enclosed in double quotes (") Quoted strings may not cross a line boundary. A double quote inside a quoted string is escaped using a backslash (\). Other escape sequences should be supported as described above.

  • End-of-line, represented by a newline character.

  • End-of-file.

  • Error: any character not considered above.

    Comments, beginning with a semicolon (;) and continuing to end-of-line are to be considered the same as whitespace, and should not be returned as tokens.

Method

  • Write your finite state machine code and test your TokenReader on sample assembly-language files to be sure that it functions properly and returns the proper token values.

  • Use the returned values from your TokenReader to check each input line for syntax correctness. Use a finite state machine that recognizes proper syntax; processing operand expressions will be done using a finite state expression recognizer discussed in class. Do not attempt to produce output code at this point.

    Test your routines carefully to ensure that you can detect all possible syntax errors.

  • Write code to do the above step twice, carrying out a two-pass scan of the input file. Your results on the second pass should be the same as on the first pass.

  • Rewrite the first pass of your assembler to enter labels into the symbol table:

    • When you encounter a label at the beginning of a line (either with a colon or as part of an EQU directive):
      • if it is already in the symbol table, report a "multiply defined label" error;
      • if it is not in the symbol table, insert it in the symbol table and give it an UNDEF value;
    • When you encounter a label following an EXTERN directive, insert it in the symbol table and initialize it to an UNDEF value;

  • Implement operand expression evaluation; for labels, temporarily assume they have zero values with a tag value of zero. In expressions, have the '$' symbol temporarily evaluate to zero with a tag value of one.

  • Rewrite the first and second passes of your assembler so that the labels are assigned values and the location counter is properly updated as the lines are scanned.

    • Rewrite the operand expression code to use the 'real' values of labels and the '$' symbol.
    • The '$' symbol should evaluate to the value of the location counter at the beginning of the line, with a tag value of one.
    • For a label which is the target of an EQU directive, evaluate the operand expression and deposit the resulting value in the symbol table corresponding to the label along with its associated tag. Note that the expression evaluator automatically determines the tag of the expression. (If this is pass two and the label was already defined in pass 1, this step is redundant, but you might check that the value in pass two is the same as on pass one in this case.)
    • For labels defined at the beginning of a line followed by a colon (i.e., not as EQU targets), deposit the value of the location counter in the symbol table corresponding to the label, and mark the label as REL.
    • For labels that appear in an EXTERN directive, deposit an Extern value in the symbol table corresponding to the label. Use the first form of the Extern constructor.
    • Increment the location counter by one for each instruction, and by one for each operand, if there is one. For a DW directive, increment the location counter for each value in the comma-separated list or, in the case of STRINGs, by one for each character in the string. For a DS directive, increment the location counter by the value of the operand (which must have a nonnegative value with an ABS tag). Other assembly language lines do not change the location counter.

  • Collect the values of the relocation dictionary during pass two, according to the locations of any object module values that are relocatable.

  • Produce the object module during the second pass of the assembler, including the relocation dictionary. Remember that the value of the location counter at the end of pass one is the same as the text length that gets output at the beginning of the object module in pass two.

    Test the code you produce using the sxx emulator. You can do an assembly and execute in one Unix line as follows: 

    java Sax file | sxxl | sxx
    which pipes the object module output of your Sax assembler into the sxxl linker and then onto the sxx emulator.

    If you prefer, you can create separate object and executable files using the assembler and linker as follows: 

    java Sax file > file.out
sxxl file.out > file.sxx
sxx file.sxx
    In this example, file is the source file, file.out is the object file produced by the assembler, and file.sxx is the executable file generated by the linker.

  • Implement and test the ENTRY, EXTERN, and PUBLIC directives, producing appropriate object module entries.

  • Implement local labels. These are labels that begin with an @ character. Local labels are prepended with the most recently encountered non-local label for the purpose of putting them in the symbol table and are otherwise treated as ordinary labels. Local labels cannot be marked PUBLIC, EXTERN, or ENTRY (although a non-local label can be marked PUBLIC or ENTRY and EQUd to a local label).

Cautions

Some expressions will not be defined until pass two, so you will need to re-evaluate expressions in pass two. If an expression value is required (e.g., to produce object module output) in pass two and is undefined, you should regard this as an error. On the other hand, it's OK for a label to remain undefined even in pass two if it doesn't appear in an operand expression that produces object module output.

Remember that DS directives must have an absolute argument that is defined and non-negative in pass one.

The relocation dictionary must include all addresses (zero-based) that contain relocatable values. These include targets of branches as well as operand expressions that evaluate to relocatable values.

You do not need to check for expression overflow or for overflow of unsigned integer constants. Caveat emptor!

You do not need to check for object module overflow — i.e., a text length that exceeds the size of the stack machine. This will be handled by the execution engine.

Uninitialized storage (using the DS directive) can be handled by using the colon-integer feature of the object module described above.
